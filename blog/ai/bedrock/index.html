    AWS Bedrock with .NET: Getting Started

    Author(s): Ajay Kumar
    Last updated: 27 Dec 2025

    The Problem: Too Many APIs, Too Much Friction

    Imagine this: your team decides to add AI to your .NET application. You pick a model, wire up the HTTP client, handle the auth, parse the response schema—and it works. Then the product team asks to try a different model. Suddenly you have two HTTP clients, two authentication flows, two response formats, and twice the maintenance burden. Add a third model and things start to spiral.

    This is the reality that many .NET teams hit when integrating generative AI directly through vendor APIs. Each provider has its own conventions, and none of them speak fluent C#.

    ℹ️ Sound familiar?

    If you've ever copy-pasted an HttpClient setup just to talk to a different AI model, this article is for you.

    What is AWS Bedrock and Why Use It from .NET?

    AWS Bedrock is Amazon's answer to the multi-model problem. It gives you a single, consistent interface to access foundation models from providers like Anthropic Claude and Amazon Titan—without juggling separate APIs, auth tokens, or request formats for each one.

    Think of Bedrock as a universal adapter—like those laptop power converters that work worldwide. You plug in once, and voltage, plug shape, and frequency differences disappear. Similarly, your .NET code plugs into Bedrock once, and differences between Claude, Titan, and other models are handled at the API layer.

    What makes it especially compelling for .NET developers is the official AWS SDK for .NET. You can invoke powerful AI models using idiomatic C# code, plugging straight into your existing projects—no context switching, no foreign tooling. Text generation, summarization, intelligent chat—all accessible through the same familiar patterns you already use for other AWS services.

    At a high level, every Bedrock request follows this flow:

    How Bedrock Unifies Foundation Models

    Here's what abstraction buys you when calling different models:

    Aspect Direct Claude API Direct Titan API Via Bedrock
    Authentication Custom API keys AWS credentials (different service) AWS IAM (unified)
    Request Format Anthropic-specific JSON Amazon-specific JSON Bedrock abstracts API layer (formats vary slightly per model)
    Error Handling Custom error codes & retry logic Different error codes & retry logic Consistent Bedrock exceptions
    Switching Models Rewrite auth + request logic Rewrite auth + request logic Often just change model ID (minor format tweaks may apply)

    Why: The Need for Simplicity, Security, and Scale

    • Simplicity: Less manual REST/API handling compared to direct integrations. The AWS SDK for .NET handles the heavy lifting so you can focus on building features.
    • Security: AWS IAM and credentials management give you enterprise-grade access control out of the box—no custom auth layers needed.
    • Scale: Bedrock is built for production, with built-in monitoring, throttling, and cost controls that would take weeks to build yourself.

    How: Step-by-Step Guide to Using AWS Bedrock from .NET

    Now that you understand what Bedrock does and why it's valuable, let's get hands-on. In this section, we'll walk through setup, authentication, and basic model invocation. Then we'll explore how this scales in production systems.

    1. Prerequisites
    • An AWS account with Bedrock access enabled.
    • AWS credentials with permissions for Bedrock (use aws configure for the shared credentials file).
    • .NET 9 SDK or later installed.
    • The following NuGet packages:
      • AWSSDK.Bedrock
      • AWSSDK.BedrockRuntime
    2. Setting Up Your Project

    You can use a simple C# script or a full .NET project. Here's how to reference the required libraries:

    
#r "nuget: AWSSDK.Bedrock, 3.7.0"
#r "nuget: AWSSDK.BedrockRuntime, 3.7.0"

    💡 For Standard .NET Projects

    If you're working in a regular .NET project (not a script), add the packages using the CLI:

    
dotnet add package AWSSDK.Bedrock
dotnet add package AWSSDK.BedrockRuntime

    Or use the NuGet Package Manager in Visual Studio.

    3. Authenticating with AWS (Recommended)

    The easiest and most secure way for most developers is to use the AWS shared credentials file. Run aws configure and the CLI will write your credentials to ~/.aws/credentials. The AWS SDK for .NET automatically reads from this file—no extra code required. 

    
# ~/.aws/credentials
[default]
aws_access_key_id     = YOUR_ACCESS_KEY_ID
aws_secret_access_key = YOUR_SECRET_ACCESS_KEY

    To use the default profile, simply instantiate the client without passing credentials—the SDK picks it up automatically:

    
// SDK automatically resolves credentials from ~/.aws/credentials [default] profile
var bedrockClient = new AmazonBedrockClient(RegionEndpoint.USEast1);

    💡 Set Up in One Command

    Run aws configure and answer the prompts. The CLI writes the credentials file for you.

    ℹ️ Need advanced authentication or production setup?

    For IAM user credentials, EC2 instance roles, or troubleshooting, see the advanced article.

    4. Common Setup Pitfalls & Solutions

    Before diving into code, let's address the most common setup mistakes that catch developers:

    Step Common Error Fix
    AWS Configure AmazonServiceException: Could not find credentials Verify credentials file exists at ~/.aws/credentials (or %USERPROFILE%\.aws\credentials on Windows)
    Enable Model Access ResourceNotFoundException: Model not available Go to AWS Console → Bedrock → Model access and request access to the model (takes 5–15 min)
    Region Mismatch ModelNotFound exception Claude 3 Haiku is only in us-east-1 and us-west-2. Verify your region matches model availability
    NuGet Version API method not found or signature mismatch Use AWSSDK.BedrockRuntime version 3.7.0 or later to match recent Bedrock API changes
    ℹ️ Region Availability Matters

    Not all models are available in all AWS regions. When you get a ModelNotFound exception, check region first before debugging your code. Use the AWS docs or Console to verify model availability in your target region.

    5. Quick Start: Your First Model Invocation

    Now that everything is configured and common pitfalls are clear, let's invoke your first model. We'll show you how to list available models and then call Claude with a simple query.

    5a. List Available Foundation Models

    Start by discovering what models Bedrock has available in your region:

    
async Task ListAvailableModels(AmazonBedrockClient client)
{
    var request = new ListFoundationModelsRequest();
    var response = await client.ListFoundationModelsAsync(request);
    Console.WriteLine($"Found {response.ModelSummaries.Count} models:");
    foreach (var model in response.ModelSummaries.Take(10))
    {
        Console.WriteLine($"  Model ID: {model.ModelId}");
        Console.WriteLine($"  Name: {model.ModelName}");
        Console.WriteLine();
    }
}

    5b. InvokeModel vs. Converse API — Which Should You Use?

    Before invoking your first model, it's important to know that Bedrock exposes two different APIs for calling foundation models:

    API Use Case Best For
    InvokeModel Single request/response; full control over model parameters One-shot tasks, API customization, batch processing
    Converse API (newer) Multi-turn conversation; handles message history automatically Chatbots, ongoing conversations, abstracts provider differences

    For this intro article, we'll focus on InvokeModel since it gives you lower-level control and is more common in production workflows. The Converse API is typically more convenient for chatbot scenarios but comes with less granular control.

    💡 Which API should I use?

    Start with InvokeModel for control and transparency. If you find yourself managing conversation history manually, switching to Converse API can simplify things.

    5c. Invoking a Model (Anthropic Claude)

    Now let's invoke Claude to get useful output. The BedrockService class wraps the Invoke API. Here is the core logic from BedrockService.cs:

    
public async Task<string> GetMovieDetails(string movieQuery)
{
    var prompt =
        $"\n\nHuman: You are a movie information expert. Provide detailed information about the movie '{movieQuery}'. " +
        "Return the results as a JSON array of objects with fields: title, year, category, directors, actors, plot, genre, rating. " +
        "If no movie is found, return an empty JSON array.\n\nAssistant:";

    var request = new InvokeModelRequest
    {
        ModelId = ModelId, // "anthropic.claude-3-haiku-20240307-v1:0"
        ContentType = "application/json",
        Accept = "application/json",
        Body = new System.IO.MemoryStream(System.Text.Encoding.UTF8.GetBytes(
            JsonSerializer.Serialize(new
            {
                anthropic_version = "bedrock-2023-05-31",
                max_tokens = 1024,
                messages = new[]
                {
                    new { role = "user", content = prompt }
                }
            })
        ))
    };

    var response = await _client.InvokeModelAsync(request);
    var responseBody = System.Text.Encoding.UTF8.GetString(response.Body.ToArray());
    return FormatJsonString(responseBody);
}

    What's happening: You build an InvokeModelRequest with your prompt, serialize it as JSON, and send it to Bedrock. The response comes back as a stream; decode it to see Claude's answer.

    💡 Test Your Setup

    Use this simple example to verify your credentials, region, and model access all work correctly before moving to production patterns.

    6. Production Patterns: Batch Processing & Error Recovery

    Single queries are great for testing, but production systems need efficiency and resilience. Let's move beyond toy examples to a realistic scenario: batch document summarization. Imagine your SaaS platform ingests customer support tickets and needs to auto-summarize them in real-time. With a naive approach, you'd loop through tickets one-by-one—slow and expensive. Here's a production-ready approach that batches requests and handles failures gracefully.

    💡 Why This Matters

    Bedrock pricing is per-request. A naive loop over 1000 tickets will cost significantly more than a batched approach with smart error recovery. Always design for batch efficiency in production.

    Compare these two approaches:

    
// Bad: loops sequentially, costs money, slow
foreach (var ticket in tickets)
{
    string summary = await service.GetMovieDetails(ticket.Description); // Slow!
    await db.SaveAsync(summary);
}
    Code Sample #5 : ❌ Naive approach: Sequential processing (slow and expensive)

    
public async Task<List<SummarizationResult>> BatchSummarizeTickets(List<SupportTicket> tickets)
{
    var results = new List<SummarizationResult>();
    var batchSize = 10; // Process in groups to avoid overwhelming Bedrock
    var results = new ConcurrentBag<SummarizationResult>();

    for (int i = 0; i < tickets.Count; i += batchSize)
    {
        var batch = tickets.Skip(i).Take(batchSize).ToList();
        var tasks = batch.Select(async ticket =>
        {
            try
            {
                var summary = await _bedrockService.SummarizeAsync(ticket.Description);
                results.Add(new SummarizationResult { TicketId = ticket.Id, Summary = summary, Success = true });
            }
            catch (ThrottlingException)
            {
                // Backoff and retry
                await Task.Delay(TimeSpan.FromMilliseconds(new Random().Next(100, 500)));
                results.Add(new SummarizationResult { TicketId = ticket.Id, ErrorMessage = "Rate limited", Success = false });
            }
            catch (Exception ex)
            {
                results.Add(new SummarizationResult { TicketId = ticket.Id, ErrorMessage = ex.Message, Success = false });
            }
        });
        await Task.WhenAll(tasks);
    }
    return results.OrderBy(r => r.TicketId).ToList();
}
    Code Sample #6 : ✓ Production approach: Batched processing with error recovery

    Why batching? By processing tickets in groups of 10 and using Task.WhenAll, you maximize throughput while respecting Bedrock's rate limits. Error handling ensures that throttling or transient failures don't crash your entire pipeline—each ticket is tracked independently.

    Example usage:

    
var service = new BedrockService();
var tickets = new List<SupportTicket> { /* ... */ };
var results = await BatchSummarizeTickets(tickets);
var successCount = results.Count(r => r.Success);
Console.WriteLine($"Processed {results.Count} tickets, {successCount} succeeded");

    Key benefits of this approach:

    • Batching: Groups requests to balance throughput and cost
    • Error Resilience: Catches throttling and gracefully degrades
    • Concurrent Processing: Task.WhenAll processes multiple tickets in parallel (respecting throttle limits)
    • Tracking: Each result includes success/failure status for audit trails
    💡 Production Tip

    Store failed ticket IDs (from Success == false) in a retry queue. Set up a scheduled job to retry failed submissions with exponential backoff.

    ⚠️ Pitfalls to Avoid & Limitations to Know

    Before shipping Bedrock to production, understand these real-world constraints. They won't stop you, but they will catch you off guard if you're unprepared.

    Regional Availability

    Not all models are available in all AWS regions. Claude 3 Haiku and Sonnet are primarily in us-east-1 and us-west-2. Titan Text is available in more regions. Check the AWS Bedrock documentation for the latest region matrix before deploying.

    ℹ️ Real-world gotcha

    ModelNotFound errors are almost always region mismatches, not code bugs. Save debugging time by verifying region first.

    Rate Limits & Quotas

    Bedrock enforces account-specific quotas to protect infrastructure. The default is commonly around 100 requests per minute per model, though this varies by region, account age, and model. Exceed your quota and you'll get a ThrottlingException. In production, implement exponential backoff and request a higher quota from AWS Support if needed.

    ℹ️ Quota varies by account

    Contact AWS Support to check your specific quota and request increases. Don't assume the 100 req/min default applies to your account.

    Pricing impact: Bedrock charges per input + output token. A 1000-ticket batch can run approximately $5–20 depending on model and verbosity (check current pricing for exact rates). Always estimate costs before scaling.

    Model Differences Matter

    Claude 3 models have different characteristics (approximate values—check AWS pricing page for current rates):

    • Haiku (fastest, cheapest): ~300ms latency, ~$0.25 per million input tokens. Best for simple tasks.
    • Sonnet (balanced): ~800ms latency, ~$3 per million input tokens. Best for complex reasoning.
    • Token limits vary: Check model docs for current context window and output limits.
    💡 Choose the right model

    Start with Haiku for testing. Switch to Sonnet only if you need better reasoning quality. Always benchmark latency + cost in your use case.

    Model Access Approval Delays

    When you first try to use a model, Bedrock requires you to request access in the AWS Console. This usually takes 5–15 minutes but can be instant. Don't request access at 4:59 PM on Friday if you plan to demo on Monday.

    Common Errors & How to Fix Them
    Error Cause Fix
    ResourceNotFoundException Model not available or not enabled AWS Console → Bedrock → Model access. Request access and wait 5–15 min.
    ValidationException Malformed request (bad JSON structure) Verify your request JSON matches Bedrock's schema and ContentType is application/json
    AccessDeniedException IAM permissions insufficient Verify your IAM user/role has bedrock:InvokeModel permissions
    ThrottlingException Rate limit hit (100 req/min default) Implement exponential backoff. Request quota increase from AWS Support if you need higher throughput.

    7. Summary

    • What: AWS Bedrock is a managed service that abstracts vendor-specific AI APIs into a unified API interface (with C# SDK support).
    • Why: It reduces friction: unified auth (IAM), consistent error handling, and easy model switching.
    • How: Use InvokeModel for single requests, Converse API for multi-turn conversations. In production, always batch, handle throttling, and forecast costs.

    This approach lets you build intelligent, generative features into your .NET apps with clarity and control. Whether you're prototyping or going to production, AWS Bedrock and .NET make GenAI accessible and robust.

    💡 💡 If You Remember Only One Thing

    Bedrock = abstraction layer. You control: the model you invoke, the prompt you send, and the cost/quality tradeoff. Start with InvokeModel, add error handling, batch in production, and always check region availability first when things break.
